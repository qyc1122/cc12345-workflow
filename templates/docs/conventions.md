不要为了单个调用方创建共享模块。

## 组件或模块约定

- 遵循现有组件/模块风格。
- 初始化逻辑放在明确入口，不分散到无关文件。
- 页面或业务模块只处理自身流程，不重复实现基础设施能力。
- 新增公共能力前，先查找已有实现和调用方。

## 路由或入口

- 新增页面、命令、服务或任务入口时，同步补充注册配置。
- 标题、权限、参数校验、环境初始化等入口级逻辑应集中维护。
- 不在多个业务文件中重复同一入口规则。

## API 与外部服务

> 完整约束见 `docs/api.md`。

- API 函数或服务封装只描述业务语义、方法、路径和参数映射。
- 通用鉴权、签名、上下文、trace、错误提示、重试等能力应由基础封装处理。
- 不绕过项目既有请求或 SDK 封装。

## 状态管理

- 只把跨页面、跨流程、跨模块或需要持久化的状态放入全局状态。
- 修改共享状态前，先查找所有调用方。
- 避免在监听器中直接修改被监听的同一数据源。

## 样式与交互

- 复用项目已有设计系统、变量、组件和交互模式。
- 只调整任务相关样式，不做全局视觉重构。
- 移动端、桌面端、可访问性和响应式约束按项目实际情况补充。

## 依赖与构建

- 使用项目既定包管理器。
- 不主动升级依赖。
- 不主动修改构建、polyfill、转译、外部资源、external 或环境变量加载规则。
- 如果确实需要构建配置变更，先说明影响面、验证方式和回滚方式。

## 测试

- 修 bug 优先补充或执行能复现问题的测试。
- 改共享逻辑时扩大验证范围。
- 无法运行测试时，说明原因，并给出已完成的替代检查。
