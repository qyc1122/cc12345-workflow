## 请求头与上下文职责

请求层或 SDK 封装统一处理：

- 鉴权信息
- 组织/项目/环境上下文
- trace/request id
- source/channel/platform
- version/build 信息
- 签名、nonce、timestamp 等安全字段

业务页面、组件或普通 API 函数不应重复生成这些通用字段。

## API 文件约定

API 文件应只做必要映射：

- 定义语义化函数名
- 选择 HTTP 方法或 SDK 方法
- 映射路径、参数和必要选项

示例：

```js
export const fetchExample = data => {
  return GET({
    url: "/api/example",
    data,
    isLoad: false
  });
};
```

请按项目实际封装替换示例中的 `GET` 和参数格式。

## 错误处理

基础封装应统一处理：

- loading 显示/关闭
- 通用业务错误提示
- 网络错误
- 非成功 HTTP 状态
- 登录失效或权限错误

业务层只处理与当前流程强相关的错误分支。

## 业务规则边界

AI 修改业务代码时必须遵守：

- 不根据字段名推导业务规则。
- 不自行新增库存、额度、权限、订单、支付、状态流转等领域限制。
- 不改变接口调用顺序，除非任务明确要求或已证明是 bug 根因。
- 不改变幂等、重试、鉴权、签名、上下文、用户身份等规则，除非接口契约或用户明确说明。
- 涉及资金、权限、身份、库存、订单、数据删除、通知发送等高风险改动，需要更强验证。
