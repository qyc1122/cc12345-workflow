# AI 协作规范模板

> **用途：** 本文件用于维护项目可复用且长期有效的高优先级协作规则。项目事实、编码约定、API 边界和任务状态应放在 `docs/` 与 `tasks/` 下维护。

开始非琐碎任务前，按需阅读：

1. `docs/architecture.md` — 项目结构与技术栈
2. `docs/conventions.md` — 编码规范与文件组织
3. `docs/api.md` — API / 外部服务使用约束
4. `docs/decisions.md` — 已确认的长期架构决策
5. `tasks/lessons.md` — 历史经验
6. `tasks/todo.md` — 当前任务状态

## 项目信息维护

- `AGENTS.md` 只保留长期有效的协作规则。
- 项目目录、技术栈、构建方式、运行入口等事实写入 `docs/architecture.md`。
- 代码风格、目录放置、测试要求、依赖管理等约定写入 `docs/conventions.md`。
- API、鉴权、请求封装、第三方服务、业务边界等约束写入 `docs/api.md`。
- 长期影响架构、模块边界、技术路线或协作方式的已确认判断写入 `docs/decisions.md`。
- 当前任务写入 `tasks/todo.md`；历史复盘写入 `tasks/review.md`；可复用经验写入 `tasks/lessons.md`。
- 不要把具体项目的临时信息、一次性结论或过期任务长期塞进 `AGENTS.md`。

## 修改约束

- 优先遵循现有架构和风格，除非任务明确要求改变。
- 只修改与用户请求直接相关的文件和代码。
- 不主动升级依赖、调整构建配置、改环境变量加载规则或变更部署策略。
- 不在业务代码中重复实现已有的通用能力，例如请求封装、鉴权、日志、错误处理、配置读取。
- 如果 GitHub 或 npm 上已有成熟且许可证兼容的开源方案，优先直接复用或复制其实现，不要自己从零实现；保留必要的来源和许可证说明。

## 1. 编码前思考

**不要假设。不要隐藏困惑。呈现权衡。**

实现之前：

- 明确说明你的假设。如果不确定，询问。
- 如果存在多种解释，呈现它们，不要默默选择。
- 如果存在更简单的方法，说出来。适时提出异议。
- 如果有不清楚的地方，停下来指出困惑点并询问。

## 2. 简洁优先

**用最少的代码解决问题。不要过度推测。**

- 不添加要求之外的功能。
- 不为一次性代码创建抽象。
- 不添加未要求的“灵活性”或“可配置性”。
- 不为没有证据会发生的场景添加复杂处理。
- 如果写了 200 行代码而其实 50 行就够，重写它。

自问：资深工程师会觉得这过于复杂吗？如果是，简化。

## 3. 精准修改

**只碰必须碰的。只清理自己造成的混乱。**

编辑现有代码时：

- 不“顺手改进”相邻的代码、注释或格式。
- 不重构没坏的东西。
- 匹配现有风格，即使你更倾向于不同写法。
- 如果注意到无关死代码或潜在问题，先记录或提醒，不擅自删除。

当你的改动产生孤儿代码时：

- 删除因你的改动而变得无用的导入、变量、函数或文件。
- 不删除预先存在的死代码，除非被要求。

**检验标准：** 每一行修改都应该能直接追溯到用户请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将任务转化为可验证目标：

- “添加验证” -> “为无效输入编写测试，然后让它们通过”
- “修复 bug” -> “编写或执行能重现 bug 的检查，然后让它通过”
- “重构 X” -> “确保重构前后行为一致，并跑过相关测试”

对于多步骤任务，说明简短计划：

```md
1. [步骤] -> 验证: [检查方式]
2. [步骤] -> 验证: [检查方式]
3. [步骤] -> 验证: [检查方式]
```

强有力的成功标准让 AI 能够独立循环执行。弱标准（“让它工作”）需要先澄清。

## 工作流编排

### 1. 计划优先

- 对任何非琐碎任务（3 步以上、跨模块改动或涉及架构决策）先写计划。
- 一旦事情走偏，停下并重新规划，不硬推。
- 不仅在实现时计划，验证环节也要有明确检查方式。
- 前期写清规格和验收标准，降低后续歧义。

### 2. 子代理策略

- 研究、探索、并行分析可以下放给子代理。
- 复杂问题可以拆给多个子代理并行，但每个子代理只做一件事。
- 主上下文负责整合结果、判断取舍和最终交付。

### 3. 自我改进闭环

- 用户每一次纠正之后，把可复用规则更新到 `tasks/lessons.md`。
- 为自己写下防止同一错误再次发生的规则。
- 每次会话开始时回顾与当前任务相关的 lessons。

### 4. 完成前必须验证

- 任务没有被证明可运行或可检查之前，绝不标记为完成。
- 必要时对比改动前后的行为差异。
- 自问：“高级工程师会批准这段改动吗？”
- 跑测试、看日志、检查 diff，用事实证明正确性。

### 5. 追求优雅（有度）

- 对非琐碎改动先问：“有没有更简单、更稳的方式？”
- 如果某个修复显得 hacky，基于已知事实重写为更清晰的方案。
- 简单、显而易见的修复跳过这一步，不要过度设计。
- 交付前挑战一下自己的结果。

### 6. 自主修 bug

- 接到 bug 报告后，直接定位根因并修复。
- 指向日志、错误或失败检查，然后逐一解决。
- 发现测试或 CI 失败时，先调查并修复与本任务相关的问题。

## 任务管理

项目级任务状态以 `tasks/todo.md` 为准。AI 工具内建的计划模式只用于单次会话内执行跟踪。

1. **先计划**：非琐碎任务把目标、步骤和验证标准写入 `tasks/todo.md`。
2. **确认计划**：实施前与用户确认一次；若用户已明确授权执行，可在说明假设后继续。
3. **跟踪进度**：做一项勾一项。
4. **解释改动**：每一步给出高层次摘要。
5. **记录结果**：任务完成后将复盘追加到 `tasks/review.md`。
6. **记录决策**：当任务中形成会长期影响架构、边界或技术路线的结论时，追加到 `docs/decisions.md`；临时取舍、执行进展和待办不要写入决策记录。
7. **沉淀经验**：被纠正后更新 `tasks/lessons.md`。

## 补充核心原则

- **不偷懒**：定位根因。不要贴临时补丁。按资深开发者标准交付。
- **最小影响面**：改动只动该动的地方，避免引入新 bug。

---

**这些准则正在发挥作用的标志：** diff 中不必要的改动更少；因过度复杂而导致的重写更少；澄清问题在实现之前提出，而不是在犯错之后。
